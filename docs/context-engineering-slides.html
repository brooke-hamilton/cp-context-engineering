What is Context Engineering?

Context engineering is the systematic practice of designing, organizing, and maintaining the information that AI systems use to understand and respond to tasks.

üí° Think of it like giving someone directions: you don't just tell them what to do (the prompt), but also provide everything they need to know to do it well (the context).

The term might sound technical, but the concept is intuitive. Every time you've organized a README file, structured a knowledge base, or decided what background information to include in a briefing, you've been doing context engineering.

Understanding Context

Context is the complete information payload provided to an LLM at inference time.

  • Instructions ‚Äî System prompts and directives
  • Knowledge ‚Äî Documentation, reference materials
  • Tools ‚Äî Functions the LLM can invoke
  • Memory ‚Äî Conversation history, state info
  • Query ‚Äî The immediate user input
  • Metadata ‚Äî File paths, timestamps, markers
Context = Assemble(instructions, knowledge, tools, memory, state, query)

Context Engineering vs. Prompt Engineering

Aspect Prompt Engineering Context Engineering
Scope Writing and optimizing instructions Curating all information accessible during inference
Components System prompts, user messages Instructions + tools + memory + data + history + metadata
Focus What to say to the LLM What information the LLM has access to
Analogy Writing a good question Preparing a comprehensive briefing packet

Core Principles

Three foundational practices for effective context engineering

Principle 1: Progressive Disclosure

Provide information incrementally, matching context depth to task complexity.

üìñ Start with essential information and expand only when needed.

Why It Works

  • Attention Budget ‚Äî Models have finite attention; position matters
  • Signal-to-Noise ‚Äî Excessive context dilutes important information
  • Cognitive Load ‚Äî Even powerful models benefit from focused context

Principle 2: Structured Note-Taking

Maintain explicit, machine-readable records that persist across interactions.

Core Practices

  • Decision Logs ‚Äî Record why choices were made
  • State Snapshots ‚Äî Capture system state at milestones
  • Context Pointers ‚Äî Reference where information lives

Example

## Decisions
- 2026-01-15: Chose REST over 
  GraphQL (simpler client 
  integration)
  
## Blocked
- Waiting on auth service

Principle 3: Attention Budget Management

Optimize how limited model attention is allocated across context.

üéØ The Core Problem

Large context windows (100K+ tokens) don't mean uniform attention. Information in the "middle" gets less attention ‚Äî the "lost in the middle" phenomenon.

Three Strategies

  • Position Optimization ‚Äî Place critical info at beginning and end
  • Context Pruning ‚Äî Remove information no longer relevant
  • Chunking & Summarization ‚Äî Load specific sections, not entire files

Pattern: ACE Framework

Intent: Prevent brevity bias and context collapse during iterative refinement.

üìä Research shows +10.6% improvement on agent benchmarks using structured updates instead of naive summarization.

Three-Phase Workflow

  1. Generation ‚Äî Create new context based on task requirements
  2. Reflection ‚Äî Evaluate what worked, what's missing, constraints
  3. Curation ‚Äî Structured update that adds detailed knowledge

ACE Framework: Example

be secure, scalable, and 
user-friendly.

Details lost!

‚úÖ Structured Updates

Security Requirements:
- OAuth2 with PKCE (iter 1)
- RBAC with 3 roles (iter 2)
- AES-256 encryption (iter 3)

Scalability Requirements:
- 100+ pods scaling (iter 1)
- DB sharding by tenant (iter 2)
- CDN for static (iter 3)

Pattern: Compaction

Intent: Manage context window limits during long-horizon tasks.

‚úÖ What to Preserve

  • Architectural decisions and rationale
  • Bugs discovered and their fixes
  • User preferences and constraints
  • Key insights from explorations

‚ùå What to Discard

  • Redundant outputs
  • Superseded work
  • Verbose explanations
  • Exploratory dead-ends

When to use: Context usage >80%, task not yet complete, clear summary possible.

Pattern: Sub-Agent Architecture

Intent: Separate concerns with specialized agents maintaining clean, focused context windows.

Architecture

Main Agent (Context: Plan + State) dispatches to:

  • Research Agent ‚Äî Context: Academic papers ‚Üí Returns summary
  • Implementation Agent ‚Äî Context: Codebase ‚Üí Returns changes
  • Test Agent ‚Äî Context: Test framework ‚Üí Returns results

Benefits

  • Clean separation ‚Äî each agent has focused, relevant context
  • Summary compaction ‚Äî sub-agents return 1-2K tokens from 10-50K explorations

Copilot: Instruction Files

Purpose: Define context once, apply to all TypeScript interactions automatically.

<instructions>
You are an expert TypeScript developer following strict type safety practices.

<rules>
- Always use explicit return types for functions
- Prefer `readonly` for immutable data structures
- Use discriminated unions for type narrowing
- Avoid `any` type; use `unknown` for truly dynamic values
</rules>

<conventions>
- Test files: `*.test.ts` co-located with source
- Error handling: Custom `AppError` class with status codes
</conventions>
</instructions>

Copilot: Progressive File Loading

Three-Tier Retrieval Strategy

Tier 1: Pre-Indexed Embeddings

Workspace structure indexed at session start ‚Äî file paths, signatures (minimal tokens)

Tier 2: Glob Patterns and Grep

file_search("**/*.ts") ‚Üí grep_search("UserRepository") (moderate tokens)

Tier 3: Targeted Read

read_file("/src/users/repository.ts", 45, 80) ‚Äî only fetch what's needed

Key Takeaways

üéØ Context engineering is about what information the AI has access to, not just what you ask it.

  • Quality over Quantity ‚Äî Aim for minimal effective context, not maximum
  • Structure Matters ‚Äî LLMs process organized information better than unstructured text
  • Position is Power ‚Äî Place critical info at beginning and end of context
  • Persist Knowledge ‚Äî Use structured notes to survive context resets
  • Progressive Loading ‚Äî Load structure first, details just-in-time
  • Measure Impact ‚Äî Track context-to-task ratio, success rates, token usage
